API Authentication Mechanisms

The method you use to prove who's calling the API. Each has a different threat profile, operational cost, and appropriate use case. Most APIs use more than one mechanism depending on the caller type.

01

API Keys

Static Shared Secrets · Most Common, Most Abused
High Risk ▼
How They Work + Where They Fail
→
What they are
A static string the client sends with every request. No expiry by default. No user context. Server looks it up in a table, grants or denies.
✗
Key in URL (Critical)
GET /data?api_key=abc123 — logged in server logs, proxy logs, browser history, referrer headers. Treat as compromised immediately.
✗
Key in source code
Committed to repos, leaked in build artifacts. GitHub searches for API keys return thousands of live credentials daily.
!
No rotation policy
A key from 3 years ago is still valid. You have no idea who has it or if it's been shared.
# BAD — key in URL GET /api/data?api_key=sk_live_abc123 # GOOD — key in header GET /api/data Authorization: ApiKey sk_live_abc123 # or X-API-Key: sk_live_abc123
Required Controls

Mandatory

✓Always in Authorization header or X-API-Key header. Never in URL query params.
✓Enforce a rotation policy. Recommended baseline: 90 days for production keys, 30 days for high-value integrations. Adjust to your risk tolerance.
✓Store in secrets vault only. CyberArk, AWS Secrets Manager, HashiCorp Vault. Never env vars in plaintext config.
✓Scope keys to minimum required permissions. A read-only key should not have write access.
✓Log and alert on key usage anomalies. Unusual volume, unusual source IPs, off-hours usage.

AI Context

!LLM integrations often use API keys to call model endpoints (OpenAI, Anthropic, Azure OpenAI). These keys have high blast radius — treat them as secrets with elevated sensitivity.
✗Never pass API keys in prompts or context. LLMs log input. Keys in prompts = keys in logs.
Risk Verdict

API keys are the weakest auth mechanism and the most widely deployed. The risk is not the key itself — it's the operational controls around it. Without rotation, scoping, vault storage, and header-only transport, API keys are effectively public credentials waiting to be found.

02

HMAC Request Signing

Hash-Based Message Authentication · Strong M2M Auth
Recommended ▼
How HMAC Signing Works
01
Build canonical request
Method + path + timestamp + body hash + selected headers. Both sides must agree on exactly what gets signed.
02
Sign with shared secret
HMAC-SHA256(canonical_string, secret_key). Produces a signature unique to this exact request at this exact time.
03
Server verifies
Recomputes the same HMAC. If it matches, the request is authentic and untampered. Timestamp check prevents replay.
Authorization: HMAC-SHA256 keyId=svc-a, timestamp=1716307200, signature=7f3a... # What gets signed: POST\n/api/v2/events\n1716307200\n x-content-sha256:abc...\n # → HMAC-SHA256 of this string with the shared secret
Key Facts

Why It's Stronger Than API Keys

✓The secret never travels on the wire. Only the signature does. Intercepting the request gives you nothing reusable.
✓Timestamp in the signature prevents replay attacks. Server rejects requests older than N seconds (typically 5 minutes).
✓Body is included in the signature. Any tampering with the payload invalidates the signature.

Where It's Used

→AWS Signature V4, Stripe webhook verification, Shopify, GitHub webhooks. Battle-tested pattern.
→Correct for service-to-service calls where both sides can share a secret securely via vault.

Limitations

!Both sides must store the shared secret securely. If the receiver is compromised, the secret is compromised.
!Does not provide non-repudiation — either party could have produced the signature. Use asymmetric signing (RS256/ES256) if you need proof of origin.
Use Case Verdict

HMAC signing is the right choice for M2M API calls where you need request integrity and replay protection without full PKI. Used correctly with vault-stored secrets and short timestamp windows, it's substantially stronger than bare API keys.

03

Service Accounts + mTLS

Certificate-Based Identity · Highest Assurance M2M
Best Practice AI/Agent ▼
Service Identity Model
→
Each service has a certificate
Issued by internal CA. Contains service identity (SPIFFE ID or CN). Short-lived — 24h to 7 days max. Auto-rotated.
→
mTLS at the connection layer
Both client and server present certs. TLS handshake validates both. Connection fails if either cert is invalid, expired, or not trusted by the CA.
→
Token binding (optional + recommended)
OAuth token bound to cert thumbprint. Stolen token can't be used without the cert that matches.
Key Facts

AI Agent Context

✓Each agent or MCP server should have its own service identity. Don't share certs across services.
✓SPIFFE/SPIRE provides workload identity for containerized agents without manual cert management.
!Operationally expensive. Requires CA, cert issuance pipeline, rotation automation, and revocation (CRL or OCSP).

When to Use

→High-value agent pipelines, regulated data access (PHI, PCI), or anywhere bearer token theft is a credible threat.
Use Case Verdict

mTLS + service accounts is the correct answer for agentic AI pipelines that handle sensitive data or have high blast radius. The operational cost is real — invest in SPIFFE/SPIRE or a service mesh to automate cert lifecycle rather than managing certs manually.

JWT Security

JWTs are used in almost every modern API auth flow. They're also widely misconfigured. These are the real attacks and the specific validation rules that prevent them.

01

Algorithm Confusion Attack

alg:none / RS256→HS256 Downgrade · Critical
Critical ▼
How The Attack Works
1
alg:none attack
Attacker sets the header alg to "none". Vulnerable libraries accept unsigned tokens because they follow the header's algorithm instruction, not a server-enforced allowlist.
2
RS256 → HS256 downgrade
Server uses RS256 (asymmetric). Attacker changes header to HS256, signs the token with the server's public key (which is often publicly available). Vulnerable servers verify with the public key as the HMAC secret.
# Vulnerable header { "alg": "none", "typ": "JWT" } # Fix: server-side allowlist, never trust header # Python example jwt.decode(token, key, algorithms=["RS256"], # explicit allowlist options={"verify_signature": True} )
Required Controls
✓Server must enforce an explicit algorithm allowlist. Never derive the algorithm from the token header.
✓Reject tokens with alg:none unconditionally. No exceptions.
✓If using asymmetric (RS256/ES256), ensure the verification key is the public key and the signing key is the private key. They are not interchangeable.
✓Prefer ES256 (ECDSA) over RS256 for new implementations. Shorter keys, same security level, harder to misuse.
Critical Verdict

This class of attack has been exploited in production systems. The fix is two lines of code — an explicit algorithm allowlist and disabled alg:none. There is no excuse for a modern production system to be vulnerable to this.

02

Missing Claim Validation

exp, aud, iss, sub · Authorization Bypass
High ▼
Claims You Must Validate
# Every JWT validation MUST check all of these: exp → token not expired (check against current time) nbf → token not used before valid time (if present) iss → issuer matches your expected auth server aud → audience matches YOUR service identifier sub → subject is a valid, active identity jti → token ID not in revocation list (if using revocation) # Common mistake: validating signature but not aud # A token issued for Service A is valid for Service B # if Service B doesn't check aud. Token substitution attack.
Key Facts
✗Signature validation alone is not enough. A token signed by a trusted IdP for a different service is still not valid for your service.
✗Missing exp check means tokens are valid forever. One leak = permanent access.
✓Validate aud against a hardcoded list of acceptable audience values for your service. Reject anything not on the list.
✓Short token lifetimes reduce exposure window. Access tokens: 15 minutes. Refresh tokens: hours to days with rotation.
!AI context: LLM services calling downstream APIs must validate the aud claim. A token issued for the LLM frontend is not valid for the backend data API.
High Risk Verdict

Most JWT libraries validate the signature by default. Most do NOT validate all claims by default. You have to explicitly configure aud, iss, and exp validation. Check your library's defaults — assume they are insufficient.

03

JWT Storage and Transport

Where You Store It Determines How It Gets Stolen
Medium ▼
Storage Options + Attack Vectors
# BAD: localStorage — XSS can read it localStorage.setItem('token', jwt) # BAD: sessionStorage — still XSS-accessible sessionStorage.setItem('token', jwt) # BETTER: HttpOnly cookie — XSS cannot read Set-Cookie: token=...; HttpOnly; Secure; SameSite=Strict # BEST for M2M: in-memory only # Store in a module-scoped variable, never persist # Refresh via client_credentials when expired
Key Facts
✗localStorage is accessible by any JavaScript on the page. One XSS vulnerability = token theft.
✓HttpOnly cookies prevent JavaScript access. Still vulnerable to CSRF — pair with SameSite=Strict and CSRF tokens.
✓M2M services should keep tokens in memory only. Fetch a fresh token via client_credentials when the current one expires. Don't persist to disk.
!AI agents running in containers: inject tokens at runtime via secrets mount, not environment variables. Env vars are visible to all processes in the container.
Storage Verdict

Where you store the token is as important as how you issue it. In-memory for M2M, HttpOnly cookie for browser apps. Never localStorage. The threat model is XSS for browsers and process inspection for services.

API Gateway Enforcement

Most security controls that live in application code get bypassed when requests hit the service directly. The gateway is the correct enforcement layer for auth, rate limiting, TLS termination, and input validation.

🏗 What the Gateway Should Own

✓TLS termination and minimum version enforcement (TLS 1.2+ minimum, TLS 1.3 preferred)
✓Auth token validation (JWT signature, claims, expiry)
✓Rate limiting and throttling per client, per endpoint
✓IP allowlisting for internal services
✓Request size limits — prevent payload floods
✓mTLS enforcement for M2M routes
✓API versioning routing and deprecation cutoffs
✓Centralized audit logging with request ID correlation

⚠ Common Gateway Misconfigurations

✗Auth validation in app code only — bypassed when gateway routes to service directly
✗TLS 1.0/1.1 still enabled — scan your gateway cipher suites
✗No request size cap — downstream services get flooded with multi-MB payloads
✗Internal services directly accessible without going through the gateway
✗Different rate limits for prod vs staging — attackers fingerprint and exploit staging
!AI endpoints (LLM inference) not behind the same gateway controls as other APIs

Rate Limiting and Throttling

Rate limiting is both an availability control and a security control. Without it, a single caller can exhaust your service, enumerate your data, or run brute-force attacks. AI endpoints have additional exposure because inference is expensive and slow.

01

Rate Limit Strategies

Fixed Window, Sliding Window, Token Bucket, Leaky Bucket
Reference ▼
Strategy Comparison
StrategyBurst HandlingUse Case
Fixed WindowEdge burstSimple quotas. Easy to implement. Vulnerable to burst at window boundary.
Sliding WindowGoodMore accurate than fixed. Good for general API rate limiting.
Token BucketAllows burstAllow controlled burst with long-term average limit. Good for variable-load APIs.
Leaky BucketNo burstSmooth output rate. Good for protecting slow downstream services.
What to Rate Limit
✓Per API key or client ID — prevent a single caller from monopolizing capacity
✓Per IP address — secondary control for unauthenticated endpoints
✓Per endpoint — your inference endpoint needs tighter limits than your health check
✓Per user (for multi-tenant AI) — prevent one tenant from degrading others
!AI-specific: rate limit on token input/output, not just request count. A single request with a 100K token prompt costs the same as 1000 normal requests.
✓Return 429 with Retry-After header. Never silently drop requests — callers need to back off correctly.
AI Endpoint Verdict

Standard request-count rate limiting is insufficient for AI inference endpoints. You need token-based limits (input + output tokens per minute) and cost-based limits in addition to request counts. A single adversarial prompt can consume more resources than thousands of normal requests.

Webhook Security

Webhooks reverse the API call direction — a third-party calls your endpoint. That means you have no control over the caller at the network level. Verification happens at the payload level.

01

HMAC Payload Verification

The Standard Webhook Security Pattern
Required ▼
Verification Pattern
# Sender (e.g., Stripe, GitHub) signs the payload: signature = HMAC-SHA256(raw_body, webhook_secret) Header: X-Signature-256: sha256={signature} # Your receiver MUST: # 1. Read raw body bytes BEFORE parsing JSON # 2. Compute expected signature # 3. Constant-time compare — NEVER string compare # WRONG — timing attack vulnerable if received_sig == expected_sig: # RIGHT — constant-time if hmac.compare_digest(received_sig, expected_sig):
Required Controls
✓Always verify the signature before processing the payload. Never process first, verify later.
✓Use constant-time comparison. String equality is vulnerable to timing attacks that reveal the signature byte by byte.
✓Validate a timestamp in the payload. Reject webhooks older than 5 minutes — prevents replay attacks.
✓Return 200 immediately, process async. Avoid webhook timeouts that trigger retries and replay edge cases.
✗Do not trust IP allowlisting alone. Vendor IPs change. Payload verification is the correct control.
!AI context: webhooks from LLM providers (model completion callbacks, fine-tune job completion) must be verified the same way. These can trigger privileged actions.
Webhook Verdict

An unverified webhook endpoint is an unauthenticated HTTP endpoint that can trigger business logic. Payload HMAC verification is the minimum required control. The two most common mistakes are string comparison instead of constant-time comparison, and processing before verifying.

CORS — Cross-Origin Resource Sharing

CORS is a browser-enforced policy. It does not protect your API from non-browser callers. Its purpose is to control which web origins can make cross-origin requests from a browser. Misconfiguration is extremely common and often has high impact.

01

CORS Misconfigurations

Origin Reflection, Wildcard + Credentials, Null Origin
Critical ▼
The Three Critical Misconfigs
# 1. ORIGIN REFLECTION (Critical) Access-Control-Allow-Origin: [whatever the request sent] # Any origin can make credentialed requests. Complete bypass. # 2. WILDCARD + CREDENTIALS (Invalid but dangerous when misconfigured) Access-Control-Allow-Origin: * Access-Control-Allow-Credentials: true # Browsers block this, but some frameworks silently fix it # to reflection, creating vuln #1 accidentally # 3. NULL ORIGIN (High) Access-Control-Allow-Origin: null # Sandboxed iframes and local files send Origin: null # Attacker can craft a page that sends null origin # CORRECT PATTERN Access-Control-Allow-Origin: https://app.yourdomain.com # Explicit allowlist. No reflection. No null. No wildcard + creds.
Required Controls
✓Maintain an explicit allowlist of permitted origins. Validate incoming Origin header against the list. Return the matching origin or nothing.
✗Never reflect the Origin header value directly into Access-Control-Allow-Origin without allowlist validation.
✗Never allow Origin: null in production. No valid production use case.
✓For public read-only APIs with no credentials: wildcard (*) is fine. For any API that sends cookies or uses Authorization headers: explicit origin required.
!CORS does not apply to server-to-server calls or curl. It is browser-only. CORS alone is not an API security control — it is a browser behavior control.
!AI apps served from a CDN calling an LLM backend: make sure the CORS policy allows the exact frontend origin. Wildcard here means any page on the internet can call your LLM endpoint from a user's browser.
CORS Verdict

Origin reflection is the most dangerous and most common CORS misconfiguration. It effectively disables CORS for credentialed requests. Check your server's CORS library defaults — many frameworks have options that enable reflection by accident. Always use an explicit allowlist.

AI-Specific API Threats

Standard API security controls are necessary but not sufficient for AI systems. These threats are specific to APIs that sit in front of or behind LLMs, agents, and ML inference endpoints.

01

Prompt Injection via API

Indirect Prompt Injection · Agent Hijacking
Critical AI-Specific ▼
Attack Pattern
1
Direct injection
Attacker sends a crafted API payload containing instructions that override the system prompt. The LLM follows the injected instruction instead of the application's intended behavior.
2
Indirect injection
Agent reads external data (email, document, web page) that contains injected instructions. The agent acts on the instructions as if they came from the user. The API call looked clean — the data it fetched was malicious.
3
Exfiltration via output
Injection instructs the LLM to include sensitive data (session tokens, API keys, user PII) in its response, which the application then returns to the attacker via the API response.
Controls
✓Treat all external data as untrusted input, including data fetched by the agent. Apply the same sanitization you'd apply to user input.
✓Separate system prompt from user-controlled input at the API call level. Use structured message formats, not string concatenation.
✓Output validation: scan LLM responses for sensitive patterns (API key formats, JWT patterns, PII) before returning to caller.
✓Principle of least privilege for agents: the agent's API credentials should only have access to what it needs for its specific task.
!No complete technical solution exists yet. Layered controls reduce risk. Defense in depth: input sanitization + output filtering + capability scoping + audit logging.
AI Threat Verdict

Prompt injection is the SQL injection of AI systems. The API layer can implement controls (input sanitization, output filtering, capability scoping) but cannot fully prevent it at the API level alone — the LLM itself has to be part of the defense. Treat it as an ongoing risk to manage, not a problem to solve once.

02

Model Endpoint Abuse

Inference Cost Attacks, Model Extraction, Data Extraction
High AI-Specific ▼
Attack Types
$
Cost exhaustion
Send maximum-token requests at high volume. Inference is expensive. Attacker runs up compute bill or degrades service for other users.
⊕
Model extraction
Systematic querying to reconstruct model behavior, fine-tuning data, or proprietary system prompts. Often done at low rate to avoid rate limiting.
⊕
Training data extraction
Specific prompts designed to cause the model to regurgitate training data, potentially including PII or proprietary content.
Controls
✓Token-based rate limits per caller. Set max input tokens and max output tokens per request and per time window.
✓Cost-based quotas per API key. Alert when a key's spend exceeds expected baseline.
✓Detect systematic enumeration patterns in access logs. Low-volume, high-diversity queries probing the same endpoint are a signal.
✓System prompt confidentiality: instruct the model not to reveal its system prompt. Also filter it from responses at the API layer.
Abuse Verdict

AI inference endpoints have a cost dimension that standard APIs don't. Standard request-count rate limiting is insufficient. You need token quotas, cost tracking, and anomaly detection on usage patterns. Treat inference budget management as a security control, not just a cost control.

03

Sensitive Data in LLM Context

Data Leakage Through Model Inputs and Outputs
High AI-Specific ▼
Leakage Vectors
✗API keys and tokens in prompts — LLM providers log inputs. Keys in prompts = keys in vendor logs.
✗PII sent to external LLM API without consent or legal basis — GDPR, HIPAA, CCPA violations depending on data type.
✗Cross-tenant data leakage in multi-user systems — one user's context bleeds into another's response when conversation history is mismanaged.
✗Verbose error responses from the LLM API revealing internal system details, file paths, or data schemas.
Controls
✓PII detection and redaction before sending to external LLM. Use Presidio, AWS Comprehend, or equivalent.
✓Use private endpoints (Azure OpenAI, AWS Bedrock) for sensitive data — traffic stays within your cloud boundary.
✓Strict conversation isolation per user session. Never share conversation history or context across users.
✓Output scanning for PII and secrets before returning LLM response to caller.
Data Leakage Verdict

Every API call to an external LLM is potentially a data transfer to a third party. Apply the same data classification controls you'd apply to any third-party data sharing: classify what's in the prompt, ensure legal basis, use private endpoints for sensitive data, and scan outputs before returning them.

API Versioning and Deprecation Security

Old API versions left running are a significant and underestimated attack surface. Security fixes applied to v3 often don't get backported to v1 and v2 that are still in production.

🔴 Why Old Versions Are Dangerous

✗Security patches applied to current version are not applied to deprecated versions
✗Old versions may have weaker auth requirements (no PKCE, no mTLS) that were upgraded in newer versions
✗BOLA/IDOR fixes in current version often leave old endpoints exposed
✗Old versions are often excluded from WAF rules and rate limiting because they're "legacy"
✗AI integrations are especially vulnerable — early AI API integrations were often built against v1 endpoints with weaker controls

✅ Deprecation Controls

✓Maintain an API version inventory. Know which versions are live, who is using them, and what their security posture is.
✓Set and enforce sunset dates. Return Deprecation and Sunset headers in every response from deprecated endpoints.
✓Apply the same gateway security controls (auth, rate limiting, WAF) to all versions, not just current.
✓Force migration deadlines. Hard cutoff with 410 Gone response, not indefinite backwards compatibility.
✓Monitor old version usage. Alert on any traffic to sunset endpoints — it's either a migration failure or an attacker who found a soft target.
