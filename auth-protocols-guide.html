AuthZ vs AuthN
A Complete Protocol Reference

AuthN = who are you  |  AuthZ = what are you allowed to do
Most protocols do one. A few do both. None of them are interchangeable.

Authentication · AuthN
"Who are you?"

Verifies identity. Establishes that the caller is who they claim to be. Produces a credential (token, assertion, cert) that downstream systems can trust. Does not say anything about what the caller can do.

Authorization · AuthZ
"What can you do?"

Controls access. Given a verified identity (or just a token), determines what resources and operations are permitted. OAuth 2.0 is pure authZ — it grants scoped access without caring who the user is.

Both Together
"Who + what?"

Full stack identity. Some protocols layer authN on top of an authZ foundation. OpenID Connect does this over OAuth 2.0. SAML bundles both in one assertion. Most real-world systems need both.

OAuth 2.0 + PKCE

RFC 6749 + RFC 7636 · Delegated Authorization
AuthZ Only Baseline Standard ▼
Authorization Code + PKCE Flow
01
User/Browser → App (Client)
login request
02
App (Client) → Auth Server
redirect + code_challenge
03
Auth Server → App (Client)
auth code
04
App (Client) → Auth Server
code + verifier
05
Auth Server → App (Client)
access token (bearer)
06
App (Client) → Resource API
bearer token
Key Facts

What it does

→Grants a client app scoped access to a resource on behalf of a user — without the user sharing their password with the app.
→PKCE adds a one-time code verifier, preventing auth code interception. Required for public clients (SPAs, mobile).

AuthN vs AuthZ

!Pure AuthZ. OAuth 2.0 alone does NOT authenticate the user. You get a token scoped to permissions — not identity proof. Add OIDC if you need identity.

AI Relevance

✓Standard for user-facing AI apps (ChatGPT, Copilot, Claude.ai). Used wherever users delegate access to a service.
✗Not designed for agent-to-agent flows where there is no user present.

Limitations

!Tokens are bearer tokens — stolen token = access. No built-in proof of sender. Mitigate with DPoP or mTLS binding.
AuthZ Verdict

OAuth 2.0 grants permissions, not identity. If your system needs to know "who is this user," layer OpenID Connect on top. Using OAuth alone and treating the access token as identity proof is a common and dangerous mistake.

OpenID Connect 1.0

Built on OAuth 2.0 · Authentication + Authorization
AuthN + AuthZ Required for Identity ▼
OIDC Flow (Adds ID Token Layer)
01
User → App (Relying Party)
login
02
App → IdP / Auth Server
OIDC auth request
03
IdP → App
ID Token + Access Token
04
App → Resource / API
validate JWT / claims
05
UserInfo Endpoint → App
sub, email, roles
Key Facts

What it adds over OAuth 2.0

→An ID Token (JWT) that proves who the user is — not just what they're allowed to do. This is the critical distinction.
→The ID token contains claims: sub (user ID), email, name, roles, session info. Signed by the IdP.

AuthN vs AuthZ

✓Does both. ID Token = AuthN (who you are). Access Token = AuthZ (what you can do). Two tokens, two jobs.

AI Relevance

✓Required when an AI system needs to assert user identity downstream — passing user context to LLMs, audit logging, or role-based access inside AI apps.
✓Used by: Okta, Entra ID, Google, Salesforce Einstein, Microsoft Copilot stack.

Limitations

!ID token is proof of auth at issuance, not continuous verification. Access token is still bearer-based without DPoP/mTLS binding.
AuthN + AuthZ Verdict

OIDC is the correct choice when you need both "who is this" and "what can they do." OAuth 2.0 alone gets you the second question only. If you're building AI workflows that need to log or act on behalf of a specific user, OIDC is the minimum viable standard.

SAML 2.0

OASIS Standard · Enterprise SSO / Browser Federation
AuthN + AuthZ Enterprise SSO Only ▼
SP-Initiated SSO Flow
01
User/Browser → Service Provider (SP)
access protected resource
02
SP → Identity Provider (IdP)
redirect + SAMLRequest (XML)
03
IdP → Browser
authenticate user
04
Browser → SP
POST SAMLResponse (signed XML)
05
SP → Browser
session granted
Key Facts

What it does

→Browser-mediated SSO using signed XML assertions. The IdP vouches for the user to the SP via the browser as a relay.
→The assertion contains both identity (who) and attributes (roles, groups) — so it covers both AuthN and limited AuthZ.

AuthN vs AuthZ

✓AuthN: signed XML assertion proves identity at the IdP. AuthZ: attribute statements carry roles/permissions the SP can use for access control.

AI Relevance

✓Acceptable for enterprise AI SaaS login (user SSO into Copilot, Einstein, etc.) where browser federation is the entry point.
✗Not a practical choice for API-level or agent calls — it is browser-bound by design. A non-browser profile (ECP) exists but is rare and not worth adopting for new work.

Limitations

!XML complexity creates real attack surface: XXE injection, signature wrapping attacks. Mature but not simple.
✗No modern API use case for new builds. Do not use for M2M or agent flows.
AuthN + AuthZ Verdict

SAML handles both AuthN and AuthZ through a single signed assertion, but it's architecturally browser-bound. It's mature and widely supported in enterprise IdPs. Use it for SSO login flows only. Never for programmatic or agentic access.

OAuth 2.0 — Client Credentials

RFC 6749 §4.4 · Machine-to-Machine Authorization
AuthZ Only M2M Standard ▼
Client Credentials Flow
01
Service / Agent
has client_id + client_secret
02
Service → Auth Server
POST credentials + scope
03
Auth Server → Service
access token
04
Service → Resource API
bearer token in request
05
Resource API → Service
200 OK
Key Facts

What it does

→No user involved. The service authenticates as itself using its own credentials and gets a scoped access token for M2M calls.

AuthN vs AuthZ

!Mostly AuthZ. The service presents a credential (which is a weak form of AuthN), but the output is a bearer token scoped to permissions — not an identity assertion.

AI Relevance

✓The go-to flow for agent-to-service calls, LLM orchestration, and automated pipelines with no user in the loop.
✓Combine with mTLS for sender-constrained tokens in high-security agent pipelines.

Limitations

!Secret management is the hard part. client_secret must be stored in a secrets vault (CyberArk, AWS Secrets Manager), never in config files or environment variables.
!Still bearer token. Stolen token = access. Pair with short TTLs and IP allowlisting where possible.
AuthZ Verdict

Client Credentials is the correct OAuth flow for M2M. No user, no browser, no redirect. The service authenticates itself and gets scoped access. The weak link is the secret — secure its storage and rotation or this flow's security collapses.

mTLS — Mutual TLS

RFC 8705 · Service-to-Service / Agent-to-Agent
AuthN Primary Best for M2M ▼
mTLS Handshake + OAuth Token Binding
01
Agent / Service A → Agent / Service B
client cert (TLS handshake)
02
Service B → Service A
server cert (TLS handshake)
03
Both sides verified → Auth Server
cert-bound token request
04
Auth Server → Service A
token bound to cert thumbprint
05
Service A → Resource API
bound token + cert
Key Facts

What it does

→Both client and server present certificates. Verifies identity at the transport layer, not just the application layer. Token is bound to the cert — stolen token alone is useless.

AuthN vs AuthZ

✓Strong AuthN at the transport layer via PKI certs. When combined with OAuth, the token binding makes it sender-constrained, adding AuthZ scope.

AI Relevance

✓Correct for MCP server auth, LLM orchestration service calls, agentic pipeline inter-service calls.
✓Used in FAPI 2.0 and open banking mandates. Eliminates bearer token theft risk at the transport layer.

Limitations

!Cert lifecycle management is operationally heavy. Requires PKI infrastructure. Not suited for user-facing browser flows.
AuthN Verdict

mTLS is the strongest available AuthN mechanism for M2M. It proves identity at the transport layer and the bound token proves the caller still holds the cert at request time. The cost is PKI complexity — you need a CA, cert issuance, rotation, and revocation infrastructure. For high-value agent pipelines, that cost is justified.

DPoP — Proof of Possession

RFC 9449 · Token Binding Without mTLS
AuthZ + Binding Emerging Best Practice ▼
DPoP Flow
01
Client / Agent
generate ephemeral keypair
02
Client → Auth Server
token request + DPoP proof JWT
03
Auth Server → Client
token bound to public key
04
Client → Resource API
token + fresh DPoP proof
05
API
verify proof matches token → granted
Key Facts

What it does

→Binds an access token to a client's private key. Every request requires a fresh signed proof — token alone is worthless without the key.

vs mTLS

→DPoP works at the application layer — no cert infrastructure needed. Lighter to implement but slightly weaker than mTLS cert binding. No PKI required.

AI Relevance

✓High value in agentic pipelines where bearer tokens pass through multiple hops. A compromised agent in the middle can't replay a stolen token.

Limitations

!Not yet universally supported by all API servers. Nonce replay protection is commonly implemented with server-side state, though stateless designs (signed or encrypted nonces) are possible. Either way it adds complexity.
AuthZ + Token Binding Verdict

DPoP is the practical middle ground between bare bearer tokens (weak) and full mTLS (operationally heavy). It gives you sender-constrained tokens without a PKI. If your environment can't do mTLS, DPoP is the right upgrade from vanilla bearer tokens.

FAPI 2.0

Financial-Grade API Profile · High-Assurance OAuth
AuthN + AuthZ Regulated Data / Healthcare ▼
FAPI 2.0 = OAuth 2.1 + PAR + mTLS/DPoP
01
Client → Auth Server
PAR: push auth request (back-channel)
02
Auth Server → Client
request_uri (short-lived ref)
03
Client → Auth Server
redirect with request_uri only
04
Auth Server → Client
mTLS or DPoP bound token
05
Client → Protected Resource
bound token + proof
Key Facts

What it is

→A security profile layered on top of OAuth 2.1. It mandates PAR, mTLS or DPoP, and short-lived tokens. Not a new protocol — a stricter ruleset.
→PAR (Pushed Auth Requests): auth parameters sent back-channel first, so they never appear in the browser URL. Prevents parameter tampering.

AuthN vs AuthZ

✓Strong AuthN through mandatory mTLS or DPoP. Strong AuthZ through sender-constrained scoped tokens. Both are enforced, not optional.

AI Relevance

✓Required for AI systems touching PHI, financial data, or regulated PII. Applies directly to healthcare and fintech contexts.

Limitations

!Significant implementation overhead. Most AI SaaS vendors do not support it natively — requires gateway or proxy enforcement.
AuthN + AuthZ Verdict

FAPI 2.0 is what you mandate when the data is sensitive enough that you can't accept weak AuthN or plain bearer tokens. It's the appropriate standard for AI systems handling financial data, health records, or regulated PII. The trade-off is implementation complexity — most vendors need a gateway in front to enforce it.

OAuth 2.0 Implicit Flow

RFC 6749 §4.2 · DEPRECATED — Do Not Use
Deprecated Legacy Only ▼
Why It Fails
01
User/Browser → Auth Server
login request
02
Auth Server → Browser
⚠ token in URL fragment
03
Token exposed in:
browser history, server logs, referrer headers
Detection Signal
If you see response_type=token in an OAuth flow, that's implicit. Flag it immediately. Replace with authorization code + PKCE.
Why It Was Replaced

Core Failures

✗Access token returned in the URL fragment. Leaks via browser history, referrer headers, and proxy logs.
✗No PKCE equivalent. No way to verify the recipient is the intended client. Auth code + PKCE solves this.
✗Still in the wild. Older AI SaaS integrations still use it. If you see response_type=token, flag it.

Replace With

✓OAuth 2.0 Authorization Code + PKCE. Drop-in replacement for SPAs and mobile apps. No server-side component required.
Deprecated Verdict

There is no valid reason to use the implicit flow in a new integration. It was deprecated because it cannot be made secure. If a vendor's documentation still shows response_type=token, their auth implementation is outdated. Require them to support authorization code + PKCE.
